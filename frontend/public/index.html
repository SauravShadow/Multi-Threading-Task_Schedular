Architecture & Design Internals

1. High Level Design (HLD) 🏢

The overarching architecture bridges a modern interactive Javascript frontend with a highly concurrent C++ backend.

Implementation Details:

The web server utilizes the cpp-httplib library to spawn a multi-threaded listener capable of handling concurrent REST invocations. When the frontend submits jobs to /api/submit, they bypass HTTP bottlenecks and are pushed instantly into an internal Task Scheduler. The user interface maintains a live, unidirectional connection reading from /api/stream using standard HTML5 Event Sources to observe execution states.

Deployment Pipeline:

This backend engine is deployed as a resilient, Zero-Downtime Windows Scheduled background task driven by a custom keep-alive script. It seamlessly auto-recovers from crashes and recompilations. External traffic is routed securely via a Cloudflare Zero-Trust Tunnel directly from task.saurav-info.xyz into the internal daemon on port 8081, safely bypassing NAT rules while joining a unified microservices network along with the developer's Web Portfolio.

2. Low Level Design (LLD) ⚙️

Beneath the HTTP handler, execution is fully asynchronous, completely independent of the frontend state.

Implementation Details:

Job tasks are encapsulated as std::packaged_task<std::string()> functors wrapping CPU-intensive brute force functions. They are stored inside a std::queue<std::function<void()>>. Meanwhile, a global JobTracker singleton meticulously records the state, hash outputs, and percent-progress into a dynamically mapped unordered_map secured by its own global mutex to allow concurrent reading for SSE distribution.

3. Custom Thread Pool 🧵

To avoid massive CPU spikes, hardware-level OS threads are instantiated only once upon scheduler startup.

Implementation Details:

The system stores a std::vector<std::thread> that continuously executes infinite lambda loops. Each thread sleeps silently via a std::condition_variable utilizing zero CPU. When the "Update Thread Pool Size" feature hits the backend, an atomic stop=true signal is dispatched, notifying all conditional variables instantly. Threads finish gracefully, are joined securely using thread.join(), and the vector is completely rebuilt.

4. Thread-Safe Blocking Queue 🚧

Ensures isolated dispatch logic where multiple producers and consumers can touch the queue simultaneously safely.

Implementation Details:

C++ std::queue is not thread safe. The codebase features a custom template wrapper that locks a std::mutex via std::lock_guard during queue.push() operations. Inside the consumer pop phase, a std::unique_lock blocks via cond_var.wait(lock, []{ return stop || !queue.empty(); }), preventing busy-waiting loops and segmentation faults!

5. Asynchronous Futures 🚀

Allows the primary HTTP listening socket to maintain massive bandwidth without blocking when complex passwords take seconds to compute.

Implementation Details:

By relying heavily on modern C++11 semantics, the TaskScheduler::submit() wrapper immediately returns a std::future<auto>. Rather than HTTP freezing during execution, the API sends a quick `202 Accepted` response. The backend logic seamlessly updates the background JobTracker, releasing the HTTP worker back to the client pool instantly.

6. Server-Sent Events (SSE) 📡

A sophisticated native streaming architecture enabling real-time visual progress without utilizing WebSockets or laggy frontend HTTP Polling.

Implementation Details:

Instead of WebSockets, the HTTP connection is kept permanently open by marking headers with Content-Type: text/event-stream. Inside a dedicated infinite loop handler, the backend uses nlohmann_json::json to actively serialize the entire internal state of the JobTracker lock map. The parsed string is emitted to the TCP stream using HTTP chunked yield transfers every exactly 250 milliseconds.
